# レビューフィードバック蓄積

PRレビューで受けた指摘を構造化し、フィードバックDBファイルに蓄積する。

## フィードバックDB

蓄積先: `/Users/shifumi.fukunaga/ghq/github.com/shifumin/my-vault/060_work/feedback-database.md`

## 処理フロー

### Step 1: PR情報の取得

ユーザーからPRのURLまたは番号を受け取り、レビューコメントを取得する。

```bash
# PRの概要
gh pr view {PR番号} --repo {owner/repo}

# レビューコメント（インライン）
gh api repos/{owner/repo}/pulls/{PR番号}/comments

# 一般コメント
gh api repos/{owner/repo}/issues/{PR番号}/comments
```

### Step 2: 指摘の抽出

取得したコメントから、PR作成者（shifumin）が他のレビュアーから受けた指摘を抽出する。
以下は除外する:
- 自分自身のコメント（セルフコメント・補足説明）
- 単なるLGTMや承認コメント
- botによる自動コメント

### Step 3: 指摘の構造化

各指摘を以下の形式で整理する:

| 項目 | 内容 |
|------|------|
| **ID** | `{カテゴリ接頭辞}-{連番3桁}`（例: `PERF-001`） |
| **タイトル** | 指摘内容を端的に表す名前 |
| **発生PR** | PR番号とリポジトリ |
| **状況** | どういうコードを書いた時に指摘されたか |
| **教訓** | 今後どうすべきか |
| **チェックポイント** | どういう状況でこの指摘を思い出すべきか |

### カテゴリとID接頭辞

| カテゴリ | ID接頭辞 | 例 |
|----------|---------|-----|
| パフォーマンス | PERF | メモリ使用量、N+1、不要なARオブジェクト生成 |
| 命名 | NAME | メソッド名、変数名、クラス名の曖昧さ |
| ドキュメント | DOC | YARD修正漏れ、コメント不足 |
| 設計 | DESIGN | 責務分離、抽象化レベル |
| テスト | TEST | テストケース不足、テストの脆さ |
| セキュリティ | SEC | 入力検証、権限チェック |
| Rails規約 | RAILS | ActiveRecordの使い方、コールバック |
| Packwerk | PACK | 依存違反、Privacy違反 |

### Step 4: フィードバックDBへの蓄積

1. フィードバックDBファイルをReadで読み込む（存在しない場合は新規作成）
2. 既存の指摘と重複しないことを確認する（同じPR・同じ内容の指摘がないか）
3. 該当カテゴリのセクションに追記する（カテゴリセクションがなければ新設）
4. IDの連番は、同カテゴリの既存最大番号 + 1 とする

#### 追記フォーマット

```markdown
### [{ID}] {タイトル}
- **発生PR**: {リポジトリ}#{PR番号}
- **状況**: {どういうコードを書いた時に指摘されたか}
- **教訓**: {今後どうすべきか}
- **チェックポイント**: {どういう状況でこの指摘を思い出すべきか}
```

### Step 5: 結果の報告

蓄積した指摘の一覧をユーザーに表示する。

```markdown
## フィードバック蓄積結果

**対象PR**: {リポジトリ}#{PR番号}
**追加件数**: {N}件

| ID | カテゴリ | タイトル |
|----|---------|---------|
| {ID} | {カテゴリ} | {タイトル} |
```

## エラー処理

| エラー | 対処 |
|--------|------|
| PRが見つからない | URLまたはPR番号の確認をユーザーに依頼 |
| レビューコメントがない | その旨を報告し、一般コメントも確認 |
| フィードバックDBファイルに書き込めない | パスの存在確認、ディレクトリ作成を試行 |
| 重複する指摘が既に存在 | 重複をスキップし、その旨を報告 |
