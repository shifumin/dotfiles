# 実装計画作成 技術仕様

planning-implementation スキルで生成するplan MDのテンプレートと設計意図。

## plan MDテンプレート

```markdown
---
created_at: {YYYY-MM-DDTHH:MM}
updated_at: {YYYY-MM-DDTHH:MM}
tags: dev/plan
project: {プロジェクト名}
target: {機能名}
status: draft
research: "[[065_dev/{research MDファイル名（拡張子なし）}]]"
---
## 概要

{何を実装するか1-2文で。目的と背景を簡潔に}

## 変更ファイル一覧

| ファイル | 変更種別 | 概要 |
|---------|---------|------|
| `path/to/file.rb` | 新規作成 | {概要} |
| `path/to/existing.rb` | 修正 | {概要} |
| `spec/path/to/file_spec.rb` | 新規作成 | {概要} |

## テンプレートの設計意図

| 要素 | 理由 |
|------|------|
| Phase/Task階層 | Claude Codeが順序通りに実装し、完了マーク(`- [x]`)を付けられる |
| ファイルパス明示 | 各タスクでどのファイルを触るか一目瞭然。実装時の探索コストを削減 |
| コードスニペット | 実装時の判断を最小化。計画フェーズで創造的判断を完了させ、実装は機械的に進める |
| 変更ファイル一覧 | 全体の影響範囲を俯瞰。レビュー時に漏れを発見しやすい |
| テストフェーズ | 最終フェーズに必ず含める。テストなしの実装完了を防ぐ |
| 検討事項 | 注釈サイクルでの議論結果を記録。実装時に「なぜこうしたか」が分かる |
| statusプロパティ | `draft` → `reviewing` → `approved` → `done` のフロー管理 |
| researchリンク | Obsidianの内部リンクでresearch MDと紐づけ。グラフビューで関連が見える |

## フロントマターのstatusフロー

| status | 意味 | 遷移タイミング |
|--------|------|---------------|
| `draft` | 初版作成済み | plan MD生成時 |
| `reviewing` | 注釈サイクル中 | ユーザーが注釈を追加し始めた時 |
| `approved` | レビュー完了、実装可能 | ユーザーが計画を承認した時 |
| `done` | 実装完了 | 全タスクのチェックボックスが完了した時 |
