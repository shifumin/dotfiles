# レビューフィードバックチェック

過去のPRレビュー指摘（フィードバックDB）に基づいて、現在の変更にセルフチェックを行う。

## フィードバックDB

参照先: `/Users/shifumi.fukunaga/ghq/github.com/shifumin/my-vault/060_work/feedback-database.md`

## 処理フロー

### Step 1: 変更差分の取得

```bash
# stagingからの差分を取得
git diff staging...HEAD

# 変更ファイル一覧
git diff staging...HEAD --name-only
```

差分が大きい場合は変更ファイル一覧を先に確認し、主要な変更ファイルに絞って分析する。

### Step 2: フィードバックDBの読み込み

フィードバックDBファイルをReadで読み込む。
ファイルが存在しない場合は「フィードバックDBがまだ作成されていません。`accumulating-review-feedback` スキルで指摘を蓄積してください。」と報告して終了する。

### Step 3: マッチング判定

変更差分の内容と、フィードバックDBの各指摘の「チェックポイント」「状況」を照合する。

判定の観点:
- **コードパターンの一致**: 指摘の「状況」に記載されたパターンが変更差分に含まれるか
- **コンテキストの類似性**: 変更の文脈が指摘の「チェックポイント」に該当するか
- **間接的な関連**: 直接一致しなくても、同じカテゴリの注意点として関連するか

判定は厳密なパターンマッチではなく、指摘の意図を理解した上での意味的な照合を行う。
誤検知（false positive）を減らすため、確信度が低い場合は「参考情報」として報告する。

### Step 4: 結果の報告

#### 該当する指摘がある場合

```markdown
## セルフレビュー結果

### 該当する過去の指摘

| ID | カテゴリ | 内容 | 対象ファイル | 該当理由 |
|----|---------|------|-------------|---------|
| {ID} | {カテゴリ} | {タイトル} | `{ファイルパス}:{行番号}` | {なぜ該当するか} |

### 各指摘の詳細

#### {ID}: {タイトル}

- **過去の教訓**: {教訓の内容}
- **今回の該当箇所**: {具体的なコード箇所の説明}
- **推奨対応**: {どう修正すべきか}
```

#### 参考情報がある場合

確信度が低いが注意した方がよい項目は「参考情報」として別セクションで報告する。

```markdown
### 参考情報（確信度: 低）

以下は直接該当するとは断言できませんが、関連する過去の指摘です:

| ID | カテゴリ | 内容 | 関連理由 |
|----|---------|------|---------|
```

#### 該当なしの場合

```markdown
## セルフレビュー結果

過去の指摘に該当するパターンは検出されませんでした。
チェックした指摘数: {N}件
```

## エラー処理

| エラー | 対処 |
|--------|------|
| フィードバックDBが存在しない | `accumulating-review-feedback` スキルでの蓄積を案内 |
| stagingブランチが存在しない | ベースブランチをユーザーに確認 |
| 差分が空 | コミットされていない変更がないか確認し、報告 |
